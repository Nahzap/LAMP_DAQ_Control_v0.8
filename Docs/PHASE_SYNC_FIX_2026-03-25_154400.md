# Fix de Sincronización de Fase en Waveforms Paralelas

**Fecha:** 2026-03-25 15:44:00  
**Componentes:** `SignalGenerator.cs`, `DataOrientedExecutionEngine.cs`  
**Severidad:** CRÍTICA — Jitter horizontal y desincronización de fase en señales paralelas

---

## 1. Problema Reportado

### Síntomas
- Dos señales waveform (500Hz y 1kHz) que inician simultáneamente muestran **drift de fase** en el osciloscopio
- El trigger no puede estabilizar la visualización porque las señales **no mantienen relación de fase constante**
- Jitter horizontal visible que indica timing inconsistente entre canales

### Evidencia en Logs
```
[DO PARALLEL] Launching 2 events in PARALLEL at 1,000000s
[DO EXEC ENGINE] Executing Waveform on PCIE-1824 CH1  ← CH1 primero
[DO EXEC ENGINE] Executing Waveform on PCIE-1824 CH0  ← CH0 después (~10-50μs delay)
[INFO] [CACHE MISS] Loading and pre-parsing LUT CSV: sine_lut.csv  ← 84ms delay!
[INFO] Signal generation started on channel 1: 1000Hz...
[INFO] Signal generation started on channel 0: 500Hz...
```

---

## 2. Diagnóstico de Causas Raíz

### Causa 1: Race Condition en Inicio Secuencial ⚠️
**Problema:** Aunque `Task.Run()` lanza tareas "en paralelo", cada tarea ejecuta `controller.StartSignalGeneration()` secuencialmente.

```csharp
// ANTES (código problemático)
parallelTasks.Add(Task.Run(() => ExecuteEventAtIndexAsync(...)));
// Dentro de ExecuteEventAtIndexAsync:
controller.StartSignalGeneration(channel, freq, amp, offset); // ← Secuencial!
```

**Resultado:** CH1 inicia 10-50μs antes que CH0, causando offset de fase permanente.

### Causa 2: Cache Miss de LUT en Primer Canal 🐌
**Problema:** El primer canal que ejecuta carga el LUT desde disco (~84ms), mientras el segundo usa cache.

```
[INFO] [CACHE MISS] Loading and pre-parsing LUT CSV: sine_lut.csv
[INFO] [CACHE LOADED] 65536 values pre-parsed to double[]  ← 84ms después
```

**Resultado:** 84ms de offset entre CH1 (primer canal) y CH0 (segundo canal).

### Causa 3: Thread.Start() Sin Sincronización 🔀
**Problema:** Cada canal inicia su thread de generación independientemente. El scheduler introduce jitter de 1-15ms.

```csharp
var thread = new Thread(() => GenerateSignal(...));
thread.Priority = ThreadPriority.Highest;
thread.Start(); // ← No garantiza simultaneidad
```

**Resultado:** Timing de inicio no determinístico entre threads.

---

## 3. Solución Implementada

### Estrategia: Barrier de Sincronización + Pre-carga de LUT

#### Paso 1: Pre-carga de LUT (Elimina Causa 2)
```csharp
// SignalGenerator.cs - Nuevo método público
public static void PreloadLutCache()
{
    lock (_lutCacheLock)
    {
        if (_cachedNormalizedValues != null) return; // Ya cargado
        
        string csvPath = SignalLUT.GetFullLutPath(SignalLUTs.SIN_LUT_FILENAME);
        var lutLines = File.ReadAllLines(csvPath);
        int size = lutLines.Length - 1;
        _cachedNormalizedValues = new double[size];
        
        // Parse completo a double[]
        for (int idx = 0; idx < size; idx++) { ... }
        _cachedLutSize = size;
    }
}
```

**Llamada:** Antes de lanzar waveforms paralelas, el execution engine llama `PreloadLutCache()` UNA VEZ.

#### Paso 2: Barrier de Sincronización (Elimina Causas 1 y 3)
```csharp
// SignalGenerator.cs - Barrier estático compartido
private static Barrier _phaseBarrier = null;

public static void PreparePhaseBarrier(int participantCount)
{
    lock (_barrierLock)
    {
        _phaseBarrier?.Dispose();
        _phaseBarrier = new Barrier(participantCount);
    }
}
```

**Uso en GenerateSignal():**
```csharp
// Cada thread espera en el barrier ANTES de iniciar generación
Barrier barrier = _phaseBarrier;
if (barrier != null)
{
    _logger.Info($"[PHASE SYNC] CH{channel} waiting at barrier...");
    barrier.SignalAndWait(1000); // Timeout 1s
    _logger.Info($"[PHASE SYNC] CH{channel} released - starting synchronized generation");
}

// AHORA todos los threads inician SIMULTÁNEAMENTE
const double TARGET_SAMPLE_RATE = 100000.0;
int samplesPerCycle = (int)(TARGET_SAMPLE_RATE / frequency);
...
```

#### Paso 3: Integración en Execution Engine
```csharp
// DataOrientedExecutionEngine.cs - ExecuteEventsAsync()

// Contar waveforms en grupo paralelo
int waveformCount = 0;
while (tempIndex < table.Count && table.StartTimesNs[tempIndex] == currentGroupStartNs)
{
    if (table.EventTypes[tempIndex] == SignalEventType.Waveform)
        waveformCount++;
    tempIndex++;
}

// Si hay 2+ waveforms paralelas: preparar sincronización
if (waveformCount > 1)
{
    System.Console.WriteLine($"[PHASE SYNC] Detected {waveformCount} parallel waveforms");
    DAQ.Services.SignalGenerator.PreloadLutCache();
    DAQ.Services.SignalGenerator.PreparePhaseBarrier(waveformCount);
}

// Lanzar tareas paralelas (ahora sincronizadas)
while (i < table.Count && table.StartTimesNs[i] == currentGroupStartNs)
{
    parallelTasks.Add(Task.Run(() => ExecuteEventAtIndexAsync(...)));
    i++;
}

await Task.WhenAll(parallelTasks);

// Limpiar barrier
if (waveformCount > 1)
{
    DAQ.Services.SignalGenerator.ClearPhaseBarrier();
    System.Console.WriteLine($"[PHASE SYNC] Barrier cleared - {waveformCount} waveforms in phase");
}
```

## 4. Flujo de Sincronización

```
Execution Engine detecta 2 waveforms paralelas en t=1.0s
    ↓
PreloadLutCache() → Carga LUT UNA VEZ (84ms)
    ↓
PreparePhaseBarrier(2) → Crea barrier para 2 participantes
    ↓
Task.Run(CH1 waveform) → Thread 1 inicia, llega al barrier, ESPERA
    ↓
Task.Run(CH0 waveform) → Thread 2 inicia, llega al barrier, ESPERA
    ↓
Barrier.SignalAndWait() → Ambos threads LIBERADOS SIMULTÁNEAMENTE
    ↓
CH1 y CH0 inician generación EN FASE (offset < 1μs)
    ↓
ClearPhaseBarrier() → Limpia barrier para próximo grupo
```

---

## 8. Impacto en Performance

- **Pre-carga de LUT:** +84ms una sola vez al inicio de grupo paralelo (vs 84ms POR CANAL antes)
- **Barrier wait:** <1ms de overhead por sincronización
- **Beneficio neto:** Eliminación completa de jitter horizontal y drift de fase

---

## 9. Notas de Implementación

- El barrier solo se activa cuando hay **2 o más waveforms paralelas**
- Waveforms individuales no tienen overhead (no usan barrier)
- Timeout de 1s en barrier previene deadlocks
- LUT cache es thread-safe (lock en `_lutCacheLock`)
- Barrier es thread-safe (lock en `_barrierLock`)
